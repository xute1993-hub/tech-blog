## 先确认部署模型

GitHub Pages 部署静态站点最关键的一点，是先确认你到底是在部署“用户主页仓库”，还是“普通项目仓库”。

如果是普通项目仓库，那么站点访问路径通常会带上仓库名。这就意味着应用必须正确设置 base path，否则静态资源和页面跳转很容易失效。

## 再确认构建输出

静态站点必须确认所有页面都能在构建阶段被导出。对于 SvelteKit 来说，这通常意味着要正确使用静态适配器，并保证页面支持预渲染。

如果某些页面依赖运行时 API，而这些 API 在静态构建阶段无法访问，就会导致构建失败或者产物不完整。

## 最后确认自动化流程

我习惯把部署流程分成几个独立步骤：

1. 安装依赖。
2. 类型检查。
3. 单元测试。
4. E2E 测试。
5. 构建产物。
6. 上传并部署。

这样做的好处是，一旦失败，问题位置会非常明确，排查效率会高很多。
